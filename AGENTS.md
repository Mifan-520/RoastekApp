## 工作流程

### AI 运行环境边界
* 当前 AI 的默认执行环境是**本机 Windows 工作区**，不是远程 Linux 服务器。
* AI 可以直接执行的范围：本地代码修改、文件整理、Docker/Compose、本地编译、本地脚本、本地验证。
* AI **不能把服务器当成本地终端直接操作**，除非用户明确提供可用的远程执行方式并允许这样做。
* 凡是服务器上的真实状态（例如服务器日志、线上容器状态、线上数据库内容、域名访问结果），默认都需要由用户提供截图、日志或远程执行结果，再继续判断。
* 涉及部署时，AI 负责：整理部署文档、检查仓库部署配置、生成/修正文档与命令、验证本地可复现流程；用户负责：在服务器实际执行、回传结果。

### AI 本地/服务器协作流程
* 默认流程：**本地 Windows 开发与验证 → 提交/同步 GitHub → 服务器拉取并部署 → 用户回传服务器结果 → AI 继续分析与修正**。
* 当任务涉及“部署到服务器”时，AI 先检查仓库根目录部署文件：`SERVER_AI_DEPLOYMENT.md`、`docker-compose.yml`、相关脚本，再给出服务器执行步骤。
* 当任务属于 MQTT 上行、前端显示、页面联调这类**可在本地 Docker 复现**的问题时，AI 默认优先使用本地 Docker 网页端口（例如 `http://127.0.0.1:8080/devices`）做全自动调试，不必先切到服务器网页端口。
* 当任务明确涉及“服务器部署结果是否生效”或用户明确要求看线上现象时，AI 才先同步部署文档与 GitHub，再由用户手动在服务器执行拉取/部署；**在用户明确回传“服务器已同步完成”之前，不能把本地 Docker 重启视为服务器已更新。**
* 只有在“需要验证服务器部署结果”且“本机代码已同步到 GitHub + 用户已完成服务器同步并回传结果”之后，AI 才进入下一轮 **结合浏览器与本地代码的回环测试/排障**。
* 当任务涉及“查看服务器是否成功”时，AI 不能臆测线上结果，必须以用户提供的服务器日志、截图、命令输出为准。
* 当任务涉及“页面显示与本机代码映射不一致”时，AI 默认采用 **本机代码 + 本地 Docker 网页端口 + 浏览器自动化** 的联合排障流程，而不是优先跳到服务器网址。
* 该联合排障流程固定为：**先读取本机代码与本地接口链路 → 打开本地 Docker 网页端口（例如 `http://127.0.0.1:8080/devices`）真实复现 → 必要时在浏览器中登录、读取 localStorage/token、直接调用本地 API → 对比“页面显示 / API 返回 / 本机代码映射”三者差异 → 再回到本机继续修正并验证**。
* 如果本地 Docker 页面可访问，AI 应优先用浏览器自动化在本地完成全自动调试；只有当用户明确要求验证线上部署结果时，才切换到服务器网址复现。
* 涉及本地页面排障时，AI 可以使用浏览器自动化读取页面文案、网络请求、登录态与接口响应；代码修改、构建、Docker 重启、测试也都默认在本机 Windows 工作区完成。
* 不允许把本地调试临时文件随意写到仓库里；如为本地验证临时创建脚本、日志或截图，使用后必须立即删除。
* 如果修改的是外部目录工程（例如仓库外的 ESP32 工程），必须只在用户指定路径内操作，不能把该工程的调试残留写回当前仓库。

### Docker 开发
* 每次修改后都要检查是否影响本地运行态；只要影响代码、配置、容器、部署或联调流程，默认立即重启 Docker 容器，并且重构前端镜像
* **逐个重启方式**：使用 `bash scripts/restart-sequential.sh`
  - 避免一次性重启导致 OpenCode 等资源冲突
  - 顺序：postgres → backend → frontend
  - 每个容器健康后才继续下一个

### 文件管理
* 创建临时脚本或截图后，使用完毕立即删除
* 不要在项目中保留用不到的文件

### GitHub 同步
* 每次大更新必须同步到 GitHub
* 推送后更新 `SERVER_AI_DEPLOYMENT.md`

### 部署目标
* 最终部署环境：阿里云服务器
* 部署流程：本地 Windows 调试 → GitHub → 服务器克隆 → Docker 部署

## 有人云 DTU 上云架构方案

### 方案 A：单片机(JSON) → 基础DTU → 云端
```
单片机 ──UART/JSON──→ 基础DTU ──MQTT/JSON──→ MQTT Broker ──MQTT──→ 后端(Node.js)
(封装JSON)            (4G透传)                              - 解析JSON
                                                          - 写入PostgreSQL
                                                               ↓
                                                          前端(React)
```

**特点**：
- 单片机自己封装 JSON
- DTU 只是透明管道，字节原样转发
- 云端收到 JSON 直接解析，无需协议转换

**适用场景**：自定义设备、有开发能力的单片机项目

---

### 方案 B：PLC → 边缘计算DTU → 云端(JSON)
```
PLC ──PLC协议(MPI)──→ 边缘DTU ──MQTT/JSON──→ MQTT Broker ──MQTT──→ 后端(Node.js)
(西门子/三菱)          (协议转换)                              - 直接存JSON
                       - PLC驱动                               - 控制下发→写寄存器
                       - 读取寄存器
                       - 封装JSON
                               ↓
                          前端(React)
```

**特点**：
- PLC 保持原有通信协议（MPI/PPI/Modbus TCP 等）
- DTU 做边缘计算：内置 PLC 协议驱动，读取寄存器，转 JSON 上传
- 云端直接收 JSON，无需关心 PLC 协议细节

**适用场景**：已有 PLC 系统，不想改动 PLC 程序，快速上云

---

### 方案 C：Modbus设备 → 基础DTU → 云端(Modbus透传)
```
Modbus设备 ──Modbus RTU──→ 基础DTU ──TCP/ModbusRTU──→ 云端Modbus网关 ──JSON──→ 后端
(传感器等)     (01 03...)      (透传)                    - 帧解析
                                                      - 虚拟串口
                                                      - 转成JSON
                                                            ↓
                                                       前端(React)
```

**特点**：
- DTU 透传原始 Modbus RTU 帧（01 03 00 00 00 0A...）
- 云端需要部署 Modbus RTU over TCP 网关
- 网关把 RTU 帧转成 JSON 后再给后端

**适用场景**：纯 Modbus 设备，云端有现成的 SCADA 软件需要直接访问设备

---

### 方案 D：Modbus设备 → 边缘计算DTU → 云端(JSON)
```
Modbus设备 ──Modbus RTU──→ 边缘DTU ──MQTT/JSON──→ MQTT Broker ──MQTT──→ 后端(Node.js)
(传感器等)   (DTU做主站)    - 定时轮询                            - 订阅telemetry
              轮询寄存器     - 解析寄存器                           - 存JSON
                           - 封装JSON {temp:25.5}                - 转发控制→写寄存器
                               ↓                                        ↓
                          MQTT Broker                              前端(React)
```

**特点**：
- DTU 做主站，主动轮询 Modbus 设备寄存器
- DTU 把寄存器数据封装成 JSON
- 云端直接收 JSON，无需 Modbus 协议栈

**适用场景**：Modbus 设备集中，希望云端简化，DTU 集中管理轮询

---

### 方案对比：后端复杂度

| 方案 | 设备类型 | DTU类型 | 上行格式 | 后端实现 | 控制下发 |
|------|----------|---------|----------|----------|----------|
| **A** | 单片机 | 基础DTU | JSON | JSON解析+存储 | MQTT直接下发 |
| **B** | PLC | 边缘DTU | JSON | JSON存储 | MQTT→DTU转PLC |
| **C** | Modbus设备 | 基础DTU | Modbus RTU | **需Modbus网关** | 网关转Modbus |
| **D** | Modbus设备 | 边缘DTU | JSON | JSON存储 | MQTT→DTU写寄存器 |

### 推荐方案

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| ESP8266自主开发，能封装JSON | **方案A** | 简单直接，云端无需额外处理 |
| 单片机只懂Modbus，不想写JSON | **方案D** | DTU做轮询，单片机保持Modbus从机 |
| 已有PLC，不想改程序 | **方案B** | DTU适配PLC协议 |
| 云端要用现成的组态软件 | **方案C** | 云端软件直接通过虚拟串口访问设备 |

### MQTT配置（后端环境变量）

```bash
# .env 文件配置
MQTT_ENABLED=true
MQTT_HOST=MQTT服务器地址
MQTT_PORT=1883
MQTT_USERNAME=                    # 可选
MQTT_PASSWORD=                    # 可选
```

### DTU设备端配置

| 参数 | 填写值 |
|------|--------|
| 服务器地址 | `8.136.109.228` |
| 端口 | `1883` |
| Client ID | `{设备ID}` (如 `SY-001`) |
| 发布主题 | `devices/{设备ID}/telemetry` |
| 订阅主题 | `devices/{设备ID}/command` |
| 数据格式 | JSON |

**说明：**
- **设备ID**：系统中的设备唯一标识，用于数据存储和MQTT主题
- **Client ID**：MQTT连接标识，可以与设备ID相同
