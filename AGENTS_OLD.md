## 工作流程

### AI 运行环境边界
* 当前 AI 的默认执行环境是**本机 Windows 工作区**，不是远程 Linux 服务器。
* AI 可以直接执行的范围：本地代码修改、文件整理、Docker/Compose、本地编译、本地脚本、本地验证。
* AI **不能把服务器当成本地终端直接操作**，除非用户明确提供可用的远程执行方式并允许这样做。
* 凡是服务器上的真实状态（例如服务器日志、线上容器状态、线上数据库内容、域名访问结果），默认都需要由用户提供截图、日志或远程执行结果，再继续判断。
* 涉及部署时，AI 负责：整理部署文档、检查仓库部署配置、生成/修正文档与命令、验证本地可复现流程；用户负责：在服务器实际执行、回传结果。

### AI 本地/服务器协作流程
* 默认流程：**本地 Windows 开发与验证 → 提交/同步 GitHub → 服务器拉取并部署 → 用户回传服务器结果 → AI 继续分析与修正**。
* 当任务涉及"部署到服务器"时，AI 先检查仓库根目录部署文件：`SERVER_AI_DEPLOYMENT.md`、`docker-compose.yml`、相关脚本，再给出服务器执行步骤。
* 当任务属于 MQTT 上行、前端显示、页面联调这类**可在本地 Docker 复现**的问题时，AI 默认优先使用本地 Docker 网页端口（例如 `http://127.0.0.1:8080/devices`）做全自动调试，不必先切到服务器网页端口。
* 当任务明确涉及"服务器部署结果是否生效"或用户明确要求看线上现象时，AI 才先同步部署文档与 GitHub，再由用户手动在服务器执行拉取/部署；**在用户明确回传"服务器已同步完成"之前，不能把本地 Docker 重启视为服务器已更新。**
* 只有在"需要验证服务器部署结果"且"本机代码已同步到 GitHub + 用户已完成服务器同步并回传结果"之后，AI 才进入下一轮 **结合浏览器与本地代码的回环测试/排障**。
* 当任务涉及"查看服务器是否成功"时，AI 不能臆测线上结果，必须以用户提供的服务器日志、截图、命令输出为准。
* 当任务涉及"页面显示与本机代码映射不一致"时，AI 默认采用 **本机代码 + 本地 Docker 网页端口 + 浏览器自动化** 的联合排障流程，而不是优先跳到服务器网址。
* 该联合排障流程固定为：**先读取本机代码与本地接口链路 → 打开本地 Docker 网页端口（例如 `http://127.0.0.1:8080/devices`）真实复现 → 必要时在浏览器中登录、读取 localStorage/token、直接调用本地 API → 对比"页面显示 / API 返回 / 本机代码映射"三者差异 → 再回到本机继续修正并验证**。
* 如果本地 Docker 页面可访问，AI 应优先用浏览器自动化在本地完成全自动调试；只有当用户明确要求验证线上部署结果时，才切换到服务器网址复现。
* 涉及本地页面排障时，AI 可以使用浏览器自动化读取页面文案、网络请求、登录态与接口响应；代码修改、构建、Docker 重启、测试也都默认在本机 Windows 工作区完成。
* 不允许把本地调试临时文件随意写到仓库里；如为本地验证临时创建脚本、日志或截图，使用后必须立即删除。
* 如果修改的是外部目录工程（例如仓库外的 ESP32 工程），必须只在用户指定路径内操作，不能把该工程的调试残留写回当前仓库。

### Docker 开发
* 每次修改后都要检查是否影响本地运行态；只要影响代码、配置、容器、部署或联调流程，默认立即重启 Docker 容器，并且重构前端镜像
* **逐个重启方式**：使用 `bash scripts/restart-sequential.sh`
  - 避免一次性重启导致 OpenCode 等资源冲突
  - 顺序：postgres → backend → frontend
  - 每个容器健康后才继续下一个

### 文件管理
* 创建临时脚本或截图后，使用完毕立即删除
* 不要在项目中保留用不到的文件

### GitHub 同步
* 每次大更新必须同步到 GitHub
* 推送后更新 `SERVER_AI_DEPLOYMENT.md`

### 部署目标
* 最终部署环境：阿里云服务器
* 部署流程：本地 Windows 调试 → GitHub → 服务器克隆 → Docker 部署

## 技术架构

### IoT 通用架构

```
单片机 → 4G DTU → MQTT Broker → 后端(Node.js) → PostgreSQL
                                              ↓
                                        REST API/WebSocket
                                              ↓
                                        前端(React HMI)
```

### 通信协议栈

| 层级 | 协议 | 说明 |
|------|------|------|
| 单片机 ↔ DTU | UART/JSON 或 Modbus RTU | 取决于设备能力 |
| DTU ↔ 服务器 | MQTT over TCP | 设备 ID 作为主题 |
| 前端 ↔ 后端 | REST API | 实时数据考虑 WebSocket |

### DTU 上云方案

#### 方案选择矩阵

| 方案 | 设备类型 | DTU 类型 | 上行格式 | 后端复杂度 | 适用场景 |
|------|----------|----------|----------|------------|----------|
| **A** | 自研单片机 | 基础 DTU | JSON | 低 | ESP 等可编程设备 |
| **B** | PLC | 边缘 DTU | JSON | 低 | 工控场景，不想改 PLC |
| **C** | Modbus 设备 | 基础 DTU | Modbus RTU | 高（需网关） | SCADA 兼容 |
| **D** | Modbus 设备 | 边缘 DTU | JSON | 低 | 传感器集中管理 |

#### 方案 A：单片机(JSON) → 基础 DTU → 云端（推荐）

```
单片机 ──UART/JSON──→ 基础DTU ──MQTT/JSON──→ MQTT Broker ──→ 后端
(封装JSON)            (4G透传)                              - 解析JSON
                                                           - 写入PostgreSQL
```

**特点**：
- 单片机自己封装 JSON
- DTU 只是透明管道，字节原样转发
- 云端收到 JSON 直接解析，无需协议转换

**适用场景**：ESP32/ESP8266 等可编程单片机
